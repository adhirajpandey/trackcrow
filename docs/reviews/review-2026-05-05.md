## Code Review Report

### Summary
TrackCrow is a Next.js App Router modular monolith for personal expense tracking. It supports Google OAuth, manual transaction CRUD, SMS-ingested transactions, dashboard analytics, custom categories/subcategories, profile token management, and an AI assistant called Crow Bot.

The codebase has moved in the right direction with service modules, route-handler tests, parser tests, and shared result contracts. The highest-risk areas are still authorization invariants around user-owned category/subcategory IDs, sensitive financial data handling, and Crow Bot's loosely typed AI/tool boundary. Recent history shows one very large service-layer/Crow Bot refactor (`1e099af`) plus smaller fixes for classifier tolerance and SMS parsing; the large refactor reduced duplication but introduced enough contract looseness that stronger tests and validation are now urgent.

Reviewed recent commits:
- `0976d9c` - focused Kotak credit-card SMS parser fix with tests; low scope, parser-specific risk.
- `1279c9b` - relaxed Crow Bot classifier schema; improved resilience but increased downstream contract risk.
- `1e099af` - large service-layer and Crow Bot refactor across 45 files; highest regression surface.
- `cb1a0e7` - transaction detail keyboard shortcuts; localized UI risk.
- `65bdd48` - dependency security bump; operationally useful, small code risk.
- `7d84b80` - initial Crow Bot conversational flow; large new AI/tool surface.
- `33f389a`, `5317d99`, `31ea835`, `037f4b6` - SMS parser and logging work; useful direction, but sensitive logging remains.

### Key Issues

#### Issue 1 - User-Owned Category And Subcategory IDs Are Not Enforced
Severity: **CRITICAL**

**Problem**  
Server actions and services accept `categoryId` and `subcategoryId` from user-controlled form data and write them without proving those IDs belong to the authenticated user. Examples: manual create passes validated IDs straight into `createManualTransaction` (`src/app/transactions/add/actions.ts:801`, `src/services/transactions/transaction-service.ts:403`), update only verifies transaction ownership before applying category/subcategory IDs (`src/services/transactions/transaction-service.ts:441`, `src/services/transactions/transaction-service.ts:488`), and preference subcategory create/edit accepts any `categoryId` while stamping the current `user_uuid` (`src/services/preferences/preferences-service.ts:520`, `src/services/preferences/preferences-service.ts:572`). The Prisma schema does not enforce a composite ownership invariant between `Subcategory.user_uuid` and `Category.user_uuid` (`prisma/schema.prisma:69`).

**Why It Matters**  
This is a real cross-user data isolation risk. A malicious authenticated user can submit another user's category/subcategory ID and attach transactions or subcategories across ownership boundaries. That corrupts analytics and can leak category names through subsequent reads.

**Suggested Fix**  
Before any write, fetch category/subcategory by `{ id, user_uuid }` and verify the subcategory belongs to the selected category. Minimal fix: add `requireOwnedCategory` and `requireOwnedSubcategory` guards and call them from add/update transaction and subcategory create/edit services.

#### Issue 2 - Raw Financial SMS Data Is Logged And Stored Without A Redaction Policy
Severity: **CRITICAL**

**Problem**  
The SMS parser logs parsed merchant/recipient metadata and logs the full unmatched message on failure (`src/common/sms-parser.ts:506`, `src/common/sms-parser.ts:521`). SMS ingestion stores the full `raw_message` on every auto transaction (`src/services/transactions/transaction-service.ts:277`). Unprocessable errors also carry `originalMessage` through service details (`src/services/transactions/transaction-service.ts:262`) and the route spreads those details into the response (`src/app/api/transactions/sms/route.ts:573`).

**Why It Matters**  
SMS messages contain amounts, merchants, account/card fragments, references, and locations. Logging or returning raw payloads expands the blast radius of every log sink and client error path. This is production-sensitive financial/PII data.

**Suggested Fix**  
Remove raw-message logging immediately. Store raw SMS only if there is a documented retention/debug requirement; otherwise store a redacted version or encrypted raw payload with retention. Never include `originalMessage` in API responses.

#### Issue 3 - SMS Ingestion Tokens Are Plaintext Bearer Secrets Without Uniqueness Or Rotation Semantics
Severity: **CRITICAL**

**Problem**  
Device ingestion tokens are stored directly as `user.lt_token` (`prisma/schema.prisma:47`), generated as raw random hex (`src/services/profile/profile-service.ts:455`), returned to the UI (`src/services/profile/profile-service.ts:451`), and looked up by plaintext token (`src/services/transactions/transaction-service.ts:249`). The schema has no `@unique` or index on `lt_token`.

**Why It Matters**  
Anyone with the token can create financial transactions for that user. Plaintext storage means a database/log exposure is immediately usable. No uniqueness/index also makes lookup slower and allows accidental duplicates.

**Suggested Fix**  
Store only a hash of the token, add `@@unique([lt_token_hash])`, and show the raw token only once at creation. Add explicit revoke/regenerate semantics and audit fields (`lastUsedAt`, `createdAt`).

#### Issue 4 - Local Workspace Contains Real Secrets In `.env`
Severity: **CRITICAL**

**Problem**  
The local `.env` file contains live-looking database, OAuth, NextAuth, and Google Generative AI secrets (`.env:1` through `.env:5`). `.env` is ignored by git (`.gitignore:16`), so this is not currently a tracked-code leak, but it is present in the working tree.

**Why It Matters**  
If this machine, terminal logs, screenshots, or future accidental commits leak `.env`, the project has immediate account/API compromise risk.

**Suggested Fix**  
Rotate these credentials and replace local `.env` sharing with `.env.example`. Add secret scanning to CI/pre-commit.

#### Issue 5 - `resetToDefaultService` Deletes User Categories Before Recreating Defaults Without A Transaction
Severity: **HIGH**

**Problem**  
`resetToDefaultService` deletes all categories for the user, then loops creating categories/subcategories one by one (`src/services/preferences/preferences-service.ts:642`, `src/services/preferences/preferences-service.ts:644`). There is no `prisma.$transaction`.

**Why It Matters**  
Any failure after the delete leaves the user with partial or empty preferences. Because transactions reference categories/subcategories, this also changes analytics categorization state unpredictably.

**Suggested Fix**  
Wrap delete + recreate in `prisma.$transaction`. Prefer nested writes or a transaction callback with all category/subcategory creation inside it.

#### Issue 6 - New User Onboarding Is Not Atomic
Severity: **HIGH**

**Problem**  
`signIn` creates the user, then separately calls `createDefaultCategoriesAndSubcategories` (`src/lib/auth.ts:89`, `src/lib/auth.ts:106`). The default seeding function performs multiple writes in a loop (`src/lib/auth.ts:27`, `src/lib/auth.ts:35`) without a transaction.

**Why It Matters**  
A transient write failure can create a valid user account with no defaults or a partial category tree. The next sign-in treats the user as existing and never retries seeding.

**Suggested Fix**  
Create the user and defaults in one transaction. If an existing user has zero categories, run an idempotent repair path.

#### Issue 7 - SMS Transaction Timestamp Uses Ingest Time, Not Event Time
Severity: **HIGH**

**Problem**  
`createSmsTransaction` always sets `timestamp` to `new Date().toISOString()` (`src/services/transactions/transaction-service.ts:274`). Parser regexes match date text but do not return a parsed transaction timestamp (`src/common/sms-parser.ts:416`, `src/common/sms-parser.ts:432`, `src/common/sms-parser.ts:447`, `src/common/sms-parser.ts:465`, `src/common/sms-parser.ts:481`).

**Why It Matters**  
Delayed SMS ingestion corrupts dashboards, monthly filters, top expense, and comparisons. For an expense tracker, transaction time is core domain data.

**Suggested Fix**  
Extend `ParsedTransactionDetails` with `timestamp?: Date | null`; parse each supported SMS date format; persist parsed event time and fall back to ingest time only when absent.

#### Issue 8 - `totalSpend` Silently Ignores Missing Category/Subcategory Filters
Severity: **HIGH**

**Problem**  
`runTotalSpend` looks up category/subcategory filters, but if the requested category or subcategory is not found, it simply does not apply that filter (`src/app/crow-bot/tools/total-spend.ts:907`, `src/app/crow-bot/tools/total-spend.ts:916`, `src/app/crow-bot/tools/total-spend.ts:932`). A query like "total spend for Rent" can return total spend for all transactions.

**Why It Matters**  
This is a silent correctness bug in a financial assistant. Users will trust a wrong number because the response says it calculated the requested total.

**Suggested Fix**  
If the user supplied a category/subcategory and no owned match exists, return `toolFail('NOT_FOUND', ...)` instead of broadening the query.

#### Issue 9 - Crow Bot Mode Contract Rejects Valid Requests When Metadata Is Missing
Severity: **HIGH**

**Problem**  
The orchestrator defaults `activeMode` to `'other'` when client metadata is missing (`src/app/crow-bot/server/chat-orchestrator.ts:154`). `checkModeMismatch` then treats `'other'` as a restrictive mode because `ALLOWED_BY_CONTEXT.other` does not exist (`src/app/crow-bot/server/intent-policy.ts:163`, `src/app/crow-bot/config/server-config.ts:307`).

**Why It Matters**  
Server behavior depends on a hidden client-only metadata contract. Any alternate client, malformed request, or regression in `useChat` metadata can cause valid intents to be rejected.

**Suggested Fix**  
Treat missing/`other` mode as "no mode restriction", or validate the request body and return a clear 400 saying mode is required.

#### Issue 10 - AI Tool Inputs Can Throw Before Tool-Level Error Mapping
Severity: **HIGH**

**Problem**  
Several tools call `schema.parse` or `new Date(...).toISOString()` outside their `try` blocks. Examples: `runExpenseComparison` parses before `try` (`src/app/crow-bot/tools/expense-comparison.ts:591`), `runTopExpense` parses before `try` (`src/app/crow-bot/tools/top-expense.ts:762`), `runTotalSpend` parses before `try` (`src/app/crow-bot/tools/total-spend.ts:882`), and `recordExpense` can throw on invalid date before validation (`src/app/crow-bot/tools/record-expense.ts:437`).

**Why It Matters**  
Model output is untrusted input. These failures bypass domain-specific `VALIDATION_ERROR` responses and become generic tool failures in the stream.

**Suggested Fix**  
Move all parsing/coercion inside `try`, use `safeParse`, and return `toolFail('VALIDATION_ERROR', ...)` with sanitized issues.

#### Issue 11 - `recordExpense` Queries Prisma With Unvalidated Category
Severity: **HIGH**

**Problem**  
`extractTransactionFields` can return `category` as `null` (`src/app/crow-bot/tools/record-expense.ts:423`, `src/app/crow-bot/tools/record-expense.ts:447`), but `runRecordExpense` queries Prisma with `name: { equals: category }` before validating category (`src/app/crow-bot/tools/record-expense.ts:478`).

**Why It Matters**  
Invalid AI output can trigger Prisma validation errors instead of a predictable "missing category" response. This is a fragile normal path for an LLM-facing feature.

**Suggested Fix**  
Add an early guard: if `category` is not a non-empty string, return `VALIDATION_ERROR` before any DB query.

#### Issue 12 - Stream Failure Messages Appear To Emit `text-delta` Without A Text Block
Severity: **HIGH**

**Problem**  
`streamToolResponse` writes `text-delta` directly on tool failure or exception (`src/app/crow-bot/utils/stream-utilities.ts:55`, `src/app/crow-bot/utils/stream-utilities.ts:71`) without a preceding `text-start`/`text-end` pair like `streamTextResponse` uses (`src/app/crow-bot/utils/stream-utilities.ts:12`).

**Why It Matters**  
If the AI SDK UI message stream protocol expects text parts to be bracketed, failure responses may render inconsistently or break clients exactly when users need clear errors.

**Suggested Fix**  
Reuse `streamTextResponse` for failures, or write a complete text part (`text-start`, `text-delta`, `text-end`) before `finish-step`.

#### Issue 13 - Query Dates Are Not Validated Before Prisma Filters
Severity: **MEDIUM**

**Problem**  
`listTransactions` directly constructs dates from query params (`src/services/transactions/transaction-service.ts:126`, `src/services/transactions/transaction-service.ts:129`). Invalid date strings become `Invalid Date` values inside Prisma filters.

**Why It Matters**  
Bad user input can become a 500 instead of a clear 400. This makes the API contract unpredictable and harder to debug.

**Suggested Fix**  
Validate date params with a small parser (`Number.isNaN(date.getTime())`) or Zod before building `where`; return `VALIDATION_ERROR` and map to 400.

#### Issue 14 - Schema Has No Indexes For The Actual Query Patterns
Severity: **MEDIUM**

**Problem**  
The Prisma schema has uniqueness constraints but no indexes on common filters/sorts like `Transaction.user_uuid + timestamp`, `categoryId`, `subcategoryId`, `recipient`, `User.lt_token`, or category/subcategory ownership lookups (`prisma/schema.prisma:11`, `prisma/schema.prisma:37`, `prisma/schema.prisma:55`, `prisma/schema.prisma:69`). Code frequently queries these fields (`src/services/transactions/transaction-service.ts:143`, `src/services/transactions/transaction-service.ts:320`, `src/services/users/user-service.ts:24`).

**Why It Matters**  
Dashboard, transaction list, SMS auth, and suggestion APIs will degrade as data grows. This is a realistic production landmine, not a hypothetical micro-optimization.

**Suggested Fix**  
Add targeted indexes: `@@index([user_uuid, timestamp])`, `@@index([user_uuid, recipient])`, `@@index([user_uuid, categoryId])`, `@@index([user_uuid, subcategoryId])`, and a unique/hash index for SMS token lookup.

#### Issue 15 - Dashboard Fetches And Serializes Full Transaction Rows For Derived Views
Severity: **MEDIUM**

**Problem**  
`DashboardPage` fetches user transactions and categories sequentially (`src/app/dashboard/page.tsx:43`, `src/app/dashboard/page.tsx:49`), then passes the full transaction array into multiple components that recompute summaries (`src/app/dashboard/page.tsx:85`, `src/app/dashboard/page.tsx:92`, `src/app/dashboard/page.tsx:101`).

**Why It Matters**  
The dashboard cost grows with every transaction and serializes more data to client components than needed. This will become visible for active users.

**Suggested Fix**  
Add a dashboard service that returns small view models: summary totals, category breakdown, timeline series, tracked preview, and untracked preview. Fetch independent user/category data in parallel where still needed.

#### Issue 18 - Type Safety Is Disabled Where It Matters Most

#### Issue 19 - Logger Has No Redaction Or Structured Sink
Severity: **MEDIUM**

**Problem**  
The logger stringifies arbitrary data (`src/lib/logger.ts:152`, `src/lib/logger.ts:157`) and routes/services pass user UUIDs, emails, raw text previews, validation errors, and parser metadata into logs (`src/lib/auth.ts:69`, `src/app/crow-bot/server/classifier.ts:44`, `src/services/transactions/transaction-service.ts:132`).

**Why It Matters**  
Logging is cross-cutting infrastructure. Without field redaction, future debug logs can leak financial or auth data even after the SMS parser logs are fixed.

**Suggested Fix**  
Add a redaction layer for keys like token, authorization, email, message, rawText, raw_message, recipient, account, reference. Prefer structured JSON logs with a safe allowlist.

#### Issue 23 - HDFC Single-Line Parser Has A Fragile Recipient Regex
Severity: **MEDIUM**

**Problem**  
The HDFC single-line parser captures recipient with `(?<recipient>[^O]+?)` (`src/common/sms-parser.ts:481`). That stops at the uppercase letter `O`, not at the `" On "` delimiter.

**Why It Matters**  
Recipients containing `O` can parse incorrectly or fail. SMS parsing is core ingestion logic, and this is a concrete regex bug.

**Suggested Fix**  
Replace with a delimiter-aware pattern such as `(?<recipient>.+?)\s+On\s+...` and add regression tests with recipients containing `O`.

#### Issue 25 - Chat Route Does Not Validate Request Shape

#### Issue 26 - Server Action Validation Does Not Normalize Duplicate/Foreign-Key Errors
Severity: **MEDIUM**

**Problem**  
Preference and transaction services catch all Prisma errors and return `INTERNAL_ERROR` (`src/services/preferences/preferences-service.ts:506`, `src/services/transactions/transaction-service.ts:417`). Unique constraint violations and invalid FK inputs become generic user messages.

**Suggested Fix**

#### Issue 27 - `getOrCreateTokenService` Has A Race That Can Return A Immediately Stale Token

**Problem**

### Final Assessment

**Codebase Maturity:**  
Mid

**Biggest Architectural Risk:**  
Ownership and validation are not enforced at the domain boundary. The UI often supplies IDs, services trust them, and the database schema does not encode enough cross-user invariants. This is the main correctness/security risk.

**Highest-Impact Next Improvement:**  
Centralize domain write guards for user-owned categories, subcategories, and transactions, then add service/API tests proving cross-user IDs are rejected.
